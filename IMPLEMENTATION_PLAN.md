#### Stage 1 вЂ” Authentication

**Files created/modified:**

**Comments will explain:**

#### Stage 2 вЂ” Link Shortening + Base62

**Files created/modified:**
- `src/lib/base62.ts` вЂ” `encode(n: number): string` implementing the actual modulo/lookup-table algorithm with step-by-step comments
- `src/app/api/links/route.ts` вЂ” POST handler: validate URL, check custom alias, generate short code via `encode(link.id)`, return short URL

**Comments will explain:**

- Why using the auto-increment ID avoids collisions

**Checkpoint:** POST `/api/links` returns `{ shortCode, shortUrl, longUrl }`; row visible in DB

#### Stage 3 вЂ” Redirect + Redis Cache-Aside

**Files created/modified:**
- `src/app/[shortCode]/page.tsx` вЂ” Next.js server component that does `redirect()` (301)
- Click logging via a fire-and-forget pattern (no `await` on the DB write so redirect is not blocked)

**Cache logic (in the page/route handler):**
1. `redis.get(shortCode)` в†’ if hit, redirect immediately
2. On miss: query Postgres, `redis.set(shortCode, longUrl, 'EX', 86400)`, redirect
3. Log click without awaiting

**Comments will explain:**
- Cache-aside pattern vs write-through
- Why we use TTL (stale data tradeoff)
- Fire-and-forget vs queue вЂ” explicit note in comments that a production system would use a job queue (BullMQ/Kafka) to guarantee delivery; fire-and-forget is used here for simplicity

**Checkpoint:** First hit в†’ miss in Redis, second hit в†’ hit in Redis (verified via `redis-cli monitor`)

#### Stage 4 вЂ” Rate Limiting

**Files created/modified:**
- `src/lib/rateLimit.ts` вЂ” fixed-window counter in Redis: `INCR user:{userId}:links:count` + `EXPIRE` on first write; returns `{ allowed: boolean, remaining: number, resetAt: number }`
- Applied in `POST /api/links` before creating the link

**Comments will explain:**
- Fixed-window algorithm: simple, slightly bursty at window boundaries
- Why token bucket / sliding window are more accurate (with brief description)
- Why Redis (not in-memory) is used: survives restarts, works across multiple server instances

**Checkpoint:** 11th POST in an hour returns `429 { error: "Rate limit exceeded", resetAt: ... }`

#### Stage 5 вЂ” Click Analytics

**Files created/modified:**
- `src/lib/analytics.ts` вЂ” parse `user-agent` header with `ua-parser-js` в†’ device type string; note on geo-IP: free without a paid API is unreliable, decision is to skip it and log `null`, explained in comments
- `src/app/api/links/[id]/analytics/route.ts` вЂ” raw Prisma `$queryRaw` with `GROUP BY date_trunc('day', clicked_at)` for daily buckets; GROUP BY referrer for top referrers; GROUP BY device_type

**The SQL query (raw for educational value):**
```sql
SELECT
  date_trunc('day', clicked_at) AS day,
  COUNT(*) AS clicks
FROM clicks
WHERE link_id = $1
  AND clicked_at >= NOW() - INTERVAL '30 days'
GROUP BY day
ORDER BY day ASC;
```

**Comments will explain:**
- What `date_trunc` does
- Why GROUP BY + aggregation in SQL is better than JS reduce
- Why `link_id` needs an index

**Checkpoint:** Analytics endpoint returns `{ totalClicks, dailyBuckets, topReferrers, deviceBreakdown }`

## Open Questions / Decisions

> [!IMPORTANT]
> **Redirect mechanism** вЂ” Next.js App Router server components can issue 301s via `redirect()` from `next/navigation`. However, for a cleaner separation of concerns I can put the redirect logic in an API route at `GET /api/r/[shortCode]` and have the `[shortCode]` page just call it, OR keep it in a server component. I'll use the server component approach (cleaner URL, no extra hop) unless you prefer otherwise.

> [!IMPORTANT]
> **Geo-IP** вЂ” Free geo-IP without a paid API is unreliable. Plan is to skip it, log `null` for country, and explain the decision with a note pointing to MaxMind GeoLite2 as the production solution. **Confirm: is this acceptable?**

> [!NOTE]
> **301 vs 302** вЂ” I'll use 302 (temporary) for redirects by default. Using 301 (permanent) causes browsers to cache the destination forever, which breaks analytics on repeat visits and makes it impossible to update the target URL. This will be explained in the notes. Let me know if you want 301 for a specific reason.

> [!NOTE]
> **Custom alias collisions** вЂ” If a user requests a custom alias that already exists, the API will return 409 Conflict with a clear error message.

> [!NOTE]
> **JWT storage** вЂ” I'll store the JWT in an `httpOnly` cookie (not localStorage) for the dashboard UI. This is the more secure approach. The API also accepts `Authorization: Bearer` headers for programmatic use.
